# Claude Code Game Studios 项目使用说明

## 1. 项目根目录与项目定位

- 项目根目录：`D:\Github\ClaudeCode-Game-Studios`
- 这是一个给 **Claude Code** 使用的“游戏工作室协作模板”，不是现成的游戏项目源码。
- 它的核心价值不是直接提供游戏功能，而是提供一整套：
  - Agent 组织架构
  - Slash Commands 工作流
  - 自动 Hook
  - 路径规则与文档模板
  - 面向 Godot / Unity / Unreal 的引擎参考资料结构

一句话理解：

> 这个仓库的作用，是把一次 Claude Code 会话组织成“一个有导演、有部门负责人、有专业岗位成员”的虚拟游戏工作室。

---

## 2. 项目里实际包含什么

### 2.1 核心组成

根据仓库内文件，当前项目主要由以下部分组成：

- `CLAUDE.md`
  - Claude Code 的主配置入口
  - 定义技术栈占位、协作原则、引用的辅助文档
- `.claude/agents/`
  - 48 个 Agent 定义文件
  - 每个 Agent 都有职责、边界、升级路径、可用工具
- `.claude/skills/`
  - 37 个 Slash Command 技能
  - 如 `/start`、`/setup-engine`、`/map-systems`、`/design-system`、`/team-*`
- `.claude/hooks/`
  - 会话启动、结束、压缩前、提交前、推送前、写文件后、子 Agent 启动时的自动脚本
- `.claude/rules/`
  - 不同目录的编码规范或文档规范
- `.claude/docs/`
  - 快速开始、Agent 名册、协作地图、上下文管理、模板等说明文档
- `docs/WORKFLOW-GUIDE.md`
  - 从想法到上线的完整工作流
- `docs/COLLABORATIVE-DESIGN-PRINCIPLE.md`
  - 协作设计原则，强调“用户决策，Agent 辅助”
- `docs/engine-reference/`
  - 针对 Godot / Unity / Unreal 的版本与模块参考文档
- `production/session-state/`
  - 保存会话中间状态，帮助恢复上下文

### 2.2 这个仓库当前“不包含”什么

当前仓库本身几乎没有游戏源码；它更像一个“工作室操作系统”模板。

也就是说，你通常会把它作为：

- 新游戏项目的模板仓库
- 已有游戏项目的 Claude Code 工作流基础设施

来使用。

## 3. 你应该怎样理解这个项目

可以把这个项目理解成四层：

### 第 1 层：总规范

- `CLAUDE.md`
- `.claude/docs/*.md`

作用：

- 告诉 Claude Code 这是一个什么项目
- 告诉 Agent 应遵守什么协作方式
- 告诉整个系统要按什么目录、什么规范工作

### 第 2 层：人员组织

- `.claude/agents/*.md`

作用：

- 定义“谁负责什么”
- 把复杂任务分给合适的角色

### 第 3 层：工作流程

- `.claude/skills/*/SKILL.md`

作用：

- 把常见工作做成可复用流程
- 例如从零开始、选引擎、拆系统、写 GDD、做团队协作、做发布

### 第 4 层：自动检查与记忆

- `.claude/hooks/*.sh`
- `.claude/settings.json`
- `production/session-state/`

作用：

- 自动检测项目缺口
- 记录当前会话状态
- 审计子 Agent 调用
- 在提交、推送、写文件前后执行检查

## 4. 新手最推荐的使用方式

### 4.1 最标准的起步流程

1. 安装 Claude Code
2. 在项目根目录启动 Claude Code
3. 先运行 `/start`
4. 按情况继续：
   - 没想法：`/brainstorm`
   - 已有概念：`/setup-engine`
   - 已有项目：`/project-stage-detect`

### 4.2 `/start` 是什么

`/start` 是入口技能，作用是“先判断你现在处于什么阶段，再推荐下一步”。

它会先静默检查：

- 有没有配置引擎
- 有没有 `design/gdd/game-concept.md`
- 有没有源码
- 有没有原型
- 有没有 GDD
- 有没有生产计划文件

然后让你从四种状态里选：

- 没有想法
- 有模糊想法
- 有清晰概念
- 已经有现成工作

最后只做一件事：

> 给你推荐下一步，而不是替你自动跑完整流程。

这点非常关键，因为本项目强调“协作，不是自动驾驶”。

## 7. 这个项目如何约束 Claude Code 的行为

## 7.1 协作原则

最重要的原则来自 `docs/COLLABORATIVE-DESIGN-PRINCIPLE.md`：

> Question -> Options -> Decision -> Draft -> Approval

也就是：

1. 先问清楚
2. 再给选项
3. 由用户做决定
4. 再出草案
5. 经批准后才写入文件

所以你不应该把它理解为“我说一句话，AI 自动做完整个游戏”。
正确理解是：

> 你是最终决策者，系统把复杂任务拆开，并安排合适的 Agent 参与。

## 7.2 Hooks 自动做什么

由 `.claude/settings.json` 配置的 hooks 会自动运行：

- `session-start.sh`
  - 启动时显示分支、最近提交、当前 sprint/milestone、活跃会话状态
- `detect-gaps.sh`
  - 检测项目缺口，如“有源码但缺设计文档”
- `validate-commit.sh`
  - 提交前检查
- `validate-push.sh`
  - 推送前提醒
- `validate-assets.sh`
  - 写入文件后检查资产相关约束
- `pre-compact.sh`
  - 压缩上下文前保存信息
- `session-stop.sh`
  - 结束时记录本次成果
- `log-agent.sh`
  - 子 Agent 启动时写审计日志

## 7.3 Rules 自动约束什么

`.claude/rules/` 负责按目录施加规则，例如：

- `gameplay-code.md`
  - 要求 gameplay 代码数据驱动、不能直接依赖 UI、要考虑 delta time
- `engine-code.md`
  - 对核心系统实现施加更严格约束
- `design-docs.md`
  - 对 GDD 文档要求完整章节
- `test-standards.md`
  - 对测试组织方式提出要求

这意味着：

> 同一个 Claude，在不同目录下工作时，会被不同规则约束。

  - 使用说明、完整工作流、引擎参考、示例会话

## 10. 最重要的使用心法

### 10.1 把它当“团队操作系统”，不要当“万能提示词包”

你不是在调用一堆孤立 Prompt，而是在用一套：

- 有分工
- 有流程
- 有升级路径
- 有自动检查
- 有上下文恢复

的工作室框架。

### 10.2 优先用技能，不要一上来直接让单个 Agent 做全套

例如：

- 想开始项目，先 `/start`
- 想拆系统，先 `/map-systems`
- 想写单个系统设计，先 `/design-system`
- 想做跨部门功能，优先 `/team-*`

这样才能真正用到这个仓库的结构化价值。

### 10.3 把设计文档放在实现前面

这个仓库的默认哲学是：

> 先有概念与 GDD，再进入实现、测试、发布。

所以如果你直接跳到编码，很多 Agent 仍然会回过头要求你先澄清设计。

### 10.4 善用 `production/session-state/active.md`

这个目录虽然当前只有 `.gitkeep`，但设计上非常重要。
很多技能会把当前系统、当前章节、下一步写进去，用来应对：

- 上下文压缩
- 会话中断
- 隔天继续

### 10.5 正确结束一个会话并开启下一个任务

这个模板现在增加了一条会话边界规则：当出现下面几种情况时，Agent 应主动建议你开启一个新对话：

- 当前任务已经完成
- 话题明显切换
- 需要切换到另一个专业角色
- 当前会话轮次过多，可能影响速度和准确性

正确的做法不是只说“新开一个聊天”，而是应该给你一个可以直接复制的新开场提示词。

标准交接格式如下：

```text
[Current task summary]
建议新开对话继续下一个任务，参考开头:
"[Suggested opening prompt with role, task, and prior context summary]"
```

这样做的好处是：

- 当前任务可以干净收尾
- 下一个任务不会和当前任务混在一起
- 你不用重新整理上下文
- 更适合跨角色切换，例如从 `gameplay-programmer` 切到 `producer`

## 11. 建议你的第一轮实际练习

如果你想最快上手，建议按这个小练习走一遍：

1. 启动 Claude Code
2. 在本仓库执行 `/start`
3. 假设你要做一个小型游戏，执行 `/brainstorm`
4. 选一个概念后执行 `/setup-engine`
5. 执行 `/map-systems`
6. 选择一个系统执行 `/design-system combat-system`
7. 执行 `/design-review`

只要你走完这 7 步，就会真正理解这个项目最核心的设计。

## 12. 结论

Claude Code Game Studios 不是游戏成品，而是一个用于游戏研发协作的 Claude Code 模板框架。

它主要解决三个问题：

1. 让 Claude Code 不再像单一聊天助手，而像分工明确的工作室团队
2. 让设计、实现、测试、发布形成结构化流程
3. 让复杂项目在多轮会话里仍然能保持上下文、审计和质量控制

如果你是第一次用，最好的起点不是直接找某个 Agent，而是：

> 从 `/start` 开始，再按技能工作流逐步推进。
