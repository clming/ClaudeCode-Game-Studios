# 文档分层说明

## 1. 这份文档解决什么问题

这个仓库里的文件很多，但它们并不处于同一个层级。

最常见的误解是：

- 以为所有 `docs/` 下的文件都会被 Agent 自动读到
- 以为所有说明文档对 Agent 的约束力都一样
- 以为只要写进仓库，运行时 Agent 就一定会执行

实际上，这个仓库里的文档更适合分成 4 层来看：

1. 运行时核心规则
2. Agent / Skill 工作文件
3. 用户学习文档
4. 项目定制参考文档

这样分层后，你就能更准确判断：

- 这个文件主要是给谁看的
- Agent 会不会默认读
- 这份文档能不能真正影响运行时行为

---

## 2. 第一层：运行时核心规则

这一层最接近“Agent 真正工作时会受其影响的内容”。

## 2.1 最核心入口

- `CLAUDE.md`
- `.claude/settings.json`

它们的地位最高。

其中：

- `CLAUDE.md`
  - 是工作室总章程
  - 会定义项目身份、技术栈、关键引用文档
- `.claude/settings.json`
  - 会定义 hooks、权限、自动保护行为

如果你想让某种原则更接近“默认生效”，通常要从这两处着手。

---

## 2.2 Agent / Skill / Rule 运行骨架

- `.claude/agents/*.md`
- `.claude/skills/*`
- `.claude/rules/*.md`

这些文件不是普通说明，而是运行时工作骨架。

### `.claude/agents/*.md`

定义：

- 岗位职责
- 委托关系
- 禁止事项
- 处理问题的方式

### `.claude/skills/*`

定义：

- 标准工作流
- 多 Agent 协作步骤
- 任务推进顺序

### `.claude/rules/*.md`

定义：

- 路径级硬性约束
- 某个目录下允许和禁止的实现模式

这些内容比普通 `docs/` 更接近“会直接影响 Agent 行为”的层。

---

## 2.3 被 `CLAUDE.md` 明确挂钩的治理文档

当前仓库里，下面这些 `.claude/docs/` 文档因为被 `CLAUDE.md` 直接引用，属于高优先级治理文档：

- `.claude/docs/directory-structure.md`
- `.claude/docs/technical-preferences.md`
- `.claude/docs/coordination-rules.md`
- `.claude/docs/coding-standards.md`
- `.claude/docs/context-management.md`

这些文件虽然在 `.claude/docs/`，但地位并不只是“用户参考”，而是更接近“运行时背景规则”。

其中：

- `.claude/docs/directory-structure.md`
  - 定义项目地图
- `.claude/docs/technical-preferences.md`
  - 定义项目技术现实
- `.claude/docs/coordination-rules.md`
  - 定义协作基本边界
- `.claude/docs/coding-standards.md`
  - 定义总编码规范
- `.claude/docs/context-management.md`
  - 定义会话、压缩、交接规则

---

## 3. 第二层：Agent / Skill 工作支撑文件

这层文件通常不会像 `CLAUDE.md` 那样自动成为最高优先级规则，但在需要时会强烈支撑 Agent 和 Skill 的工作。

主要包括：

- `.claude/docs/agent-coordination-map.md`
- `.claude/docs/agent-roster.md`
- `.claude/docs/rules-reference.md`
- `.claude/docs/review-workflow.md`
- `.claude/docs/skills-reference.md`
- `.claude/docs/setup-requirements.md`
- `.claude/docs/hooks-reference.md`

这些文件更像“工作说明书”和“辅助索引”。

### 这一层的特点

- 很重要
- 但不是所有任务开局都会自动读
- 往往在特定场景下才会被读取或引用

例如：

- 需要理解组织关系时，会更可能参考 `.claude/docs/agent-coordination-map.md`
- 需要判断某路径规则时，会更可能参考 `.claude/docs/rules-reference.md`
- 需要检查评审流程时，会更可能参考 `.claude/docs/review-workflow.md`

---

## 4. 第三层：用户学习文档

这层文件主要是给你看的，用来帮助你理解和使用这个模板。

它们非常重要，但默认不等于“运行时规则文件”。

典型文件包括：

- `README.md`
- `.claude/docs/quick-start.md`
- `docs/WORKFLOW-GUIDE.md`
- `docs/项目使用说明.md`
- `docs/快速上手清单.md`
- `docs/Agent调度流程.md`

这类文档的核心作用是：

- 帮你快速上手
- 帮你理解流程
- 帮你知道应该用哪个 Agent 或 Skill
- 帮你理解项目整体方法

### 对 Agent 的影响方式

这类文档通常不会在每次任务开始时自动生效。

它们影响 Agent 的方式通常是：

1. 你主动让 Agent 参考它
2. 更高层文件引用了它
3. 当前任务和它强相关，Agent 才会按需读取

---

## 5. 第四层：项目定制参考文档

这层文件更多是“给未来的你”或“给维护者”看的，用来帮助你定制新项目或改造工作室。

典型文件包括：

- `docs/AI工作室搭建方法.md`
- `docs/技术Agent学习.md`
- `docs/项目定制修改点说明.md`
- `docs/examples/*`

它们更偏：

- 方法论
- 改造指南
- 示例参考
- 工作室搭建经验

这类文件一般不会成为默认运行时规则，除非你把其中内容继续上提到：

- `CLAUDE.md`
- `.claude/agents/*.md`
- `.claude/rules/*.md`
- `.claude/skills/*`

---

## 6. 你刚问的两个文件，应该放在哪一层

## 6.1 `docs/COLLABORATIVE-DESIGN-PRINCIPLE.md`

它最适合归类为：

- 介于“第一层运行时核心规则”和“第三层用户学习文档”之间

原因是：

1. 它放在 `docs/` 下，形式上是说明文档
2. 但它被 `CLAUDE.md` 明确引用了
3. 它承载的是用户驱动协作原则，而不是普通教程

所以它不是一个纯“给你看的教学文档”，也不是最底层的普通参考文档。

更准确地说，它属于：

> 被总章程挂钩的高优先级原则文档。

### Agent 会不会涉及读它

会，有可能。

尤其在这些情况：

- 新会话初始化时需要理解协作原则
- 当前任务涉及审批、写文件前确认、交接边界
- 你明确要求“按协作原则来”

它不是每个任务都会被逐字读取，但它明显比普通 `docs/` 文档更接近运行时原则层。

---

## 6.2 `docs/WORKFLOW-GUIDE.md`

它最适合归类为：

- 第三层用户学习文档

原因是：

1. 当前仓库没有在 `CLAUDE.md` 里直接引用它
2. hooks 也没有读取它
3. 它主要承担“完整使用手册”和“阶段流程说明”的职责

所以它更多是：

> 给你看的全流程操作说明书。

### Agent 会不会涉及读它

默认不会优先读。

但在这些情况可能会读：

- 你明确说“请参考 `docs/WORKFLOW-GUIDE.md`”
- 当前任务是梳理整体流程、项目阶段、工作流设计
- Agent 需要从仓库中找流程型说明时，按需检索到它

所以它不是“完全与 Agent 无关”，但不是默认运行骨架的一部分。

---

## 7. 用一句话判断“Agent 会不会默认读”

可以用这个简单判断法：

## 7.1 大概率会更早进入 Agent 视野的

## 7.3 主要给你看，除非你点名要求的

- 定制说明类文档
- 项目理解类文档

这类文件很有价值，但它们更适合拿来帮助你做正确改造，而不是直接当作默认运行规则。

## 8. 对你当前最关心问题的直接结论

如果只回答你刚才的问题：

### `docs/COLLABORATIVE-DESIGN-PRINCIPLE.md`

- 不是纯粹只给你看的
- 因为它被 `CLAUDE.md` 明确引用，所以属于高优先级原则文档
- Agent 在很多涉及协作协议的场景下，很可能会受它影响

### `docs/WORKFLOW-GUIDE.md`

- 主要是给你看的
- 是全流程使用手册
- Agent 不会默认把它当作启动即生效的规则文件
- 但在流程分析任务中，可能按需读取它

## 9. 如果你希望某个 `docs/` 文件真正更强地影响 Agent

最稳的方法不是单纯把内容写在 `docs/` 里，而是把它上提或下沉到更合适的层。

常见做法有 4 种：

1. 在 `CLAUDE.md` 里显式引用它
2. 把关键原则写进 `.claude/agents/*.md`
3. 把路径级约束写进 `.claude/rules/*.md`
4. 把流程写进 `.claude/skills/*`

所以如果未来你再写新文档，可以先问自己一句：

> 我是想让它“帮助我理解”，还是想让它“更稳定地影响 Agent 行为”？

答案不同，文件应该放的层级也不同。
