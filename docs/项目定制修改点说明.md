# 项目定制修改点说明

## 1. 文档目的

这份文档说明：

- 当你的 APP 项目或游戏项目技术栈不同
- 当你的平台范围不同
- 当你的工程组织不同

应该修改这个模板的哪些地方，才能把 Claude Code Game Studios 变成一个适合该项目的专用 AI 项目工作室。

---

## 2. 总原则

不要试图让一个完全通用的模板同时完美适配所有项目。

更合理的方式是：

1. 把当前模板作为母版
2. 每当创建新项目时，复制出一个新仓库
3. 针对该项目的技术现实做定制
4. 最终形成“该项目专用 AI 工作室”

这和现实公司在新项目立项时新建工作室、本地化团队规则，本质上是同一个思路。

---

## 3. 必改文件清单

每当你要让模板适配一个新项目，优先检查并修改下面这些地方。

## 3.1 项目总纲

### `CLAUDE.md`

你应该修改：

- Engine
- Language
- Build System
- Asset Pipeline

如果项目是：

- 原生 Android/iOS APP
- Web 前端主导
- 后端服务主导

那这里不要再沿用默认的 Godot / Unity / Unreal 占位，而要改成该项目真实技术栈。

---

## 3.2 技术偏好文件

### `.claude/docs/technical-preferences.md`

你应该修改：

- 命名规范
- 语言风格
- 框架约定
- 性能预算
- 测试方式
- 禁止模式

这是“把模板变成你项目技术文化”的核心文件。

---

## 3.3 目录结构文件

### `.claude/docs/directory-structure.md`

你应该修改：

- 源码目录划分
- 前端目录
- 后端目录
- 数据目录
- 工具目录

示例：

如果项目是多端应用，可改成：

```text
src/
  mobile/
    android/
    ios/
  web/
  backend/
    api/
    services/
    db/
assets/
design/
docs/
tests/
tools/
```

---

## 3.4 Agent 文件

### `.claude/agents/*.md`

你应该修改：

- 现有 Agent 的职责范围
- 报告对象
- 协作对象
- 负责目录
- 禁止越权点

如果项目需要新增角色，也在这里新增。

---

## 3.5 规则文件

### `.claude/rules/*.md`

你应该修改或新增：

- 与目录结构对应的路径规则
- 各技术栈的编码约束
- 各平台的性能与安全约束

---

## 3.6 技能文件

### `.claude/skills/*.md`

你应该修改：

- 项目常用工作流
- 多 Agent 编排方式
- 团队协作顺序

例如：

- 如果你的项目以 Web + Backend 为主，现有 `team-ui` 就不够，需要新增 Web / Backend 协作技能

---

## 4. 按项目类型看应改哪些地方

## 5. 不同语言和框架时，要改哪些点

当项目语言和框架变化时，至少检查这 6 项：

1. 命名规范
2. 目录结构
3. 构建方式
4. 测试框架
5. 性能约束
6. 安全约束

### 例子

#### Kotlin + Android

你应该补：

- Activity / Fragment / Compose 的边界
- ViewModel / 状态管理方式
- API 调用层规范
- 协程和线程规则

#### Swift + iOS

你应该补：

- SwiftUI / UIKit 的边界
- 状态与路由规则
- 生命周期约束
- 本地存储与网络层约束

#### React / Vue Web

你应该补：

- 组件边界
- 状态管理规则
- API 层封装规则
- 路由规则
- SSR / CSR 约束

#### Go / Node / Java 后端

你应该补：

- 服务分层
- DTO / Entity / Service 边界
- API versioning
- 错误码规范
- 日志、trace、监控
- 数据库事务和 migration 规则

## 6. 如何真正创建“专用 AI 项目工作室”

建议你按以下流程操作。

## Step 1：复制模板

基于当前仓库创建一个项目副本。

## Step 2：确定项目类型

明确是：

- 游戏客户端项目
- 原生 APP 项目
- Web 项目
- 后端服务项目
- 多端混合项目

## Step 3：确定技术栈

明确：

- 编程语言
- 前端框架
- 后端框架
- 构建与部署方式

## Step 4：重写总纲

修改：

- `CLAUDE.md`
- `technical-preferences.md`

## Step 5：重组 Agent

做以下动作：

- 删除不需要的 Agent
- 保留关键治理 Agent
- 新增专用平台 Agent
- 调整汇报关系

## Step 6：重写规则

确保：

- 每个正式源码目录都有对应规则
- 每条规则都反映真实工程约束

## Step 7：重写技能

确保工作流与项目一致，例如：

- 游戏项目保留 `/team-combat`
- Web 项目新增 `/team-web`
- 后端项目新增 `/team-backend`

## Step 8：形成专属工作室

最终你得到的不是“通用模板的一份副本”，而是：

- 某个具体项目的 AI 研发工作室

这时它才真正像现实公司里“为一个新项目专门组建的工作室”。

---

## 7. 实操建议

如果你要长期使用这个体系，建议把项目定制动作分成两轮：

### 第一轮：轻量定制

只改：

- `CLAUDE.md`
- `technical-preferences.md`
- `directory-structure.md`
- 少量 Agent

适合快速启动。

### 第二轮：深度定制

再改：

- 新增 Agent
- 新增 Rules
- 新增 Skills
- 新增团队编排方式

适合项目已经明确、进入稳定研发阶段。

---

## 8. 最终结论

要让这个模板真正适合你的项目，核心不是“多写几个提示词”，而是：

1. 调整项目总纲
2. 调整技术偏好
3. 调整目录结构
4. 调整 Agent 组织
5. 调整规则体系
6. 调整工作流技能

做到这 6 步后，你的模板就不再只是“Claude Code Game Studios”，而会变成：

> 你的某个 APP 项目或游戏项目的专用 AI 项目工作室。
