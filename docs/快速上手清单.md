# Claude Code Game Studios 快速上手清单

## 1. 这份清单适合谁

如果你已经知道这个仓库是做什么的，但还不知道“第一天到底该怎么用”，就按这份清单执行。

目标：

- 用最短路径理解模板
- 在 Claude Code 里跑通第一轮流程
- 知道什么时候该用哪个技能

---

## 2. 10 分钟认识项目

先只看这 6 个文件：

1. `README.md`
2. `CLAUDE.md`
3. `.claude/docs/quick-start.md`
4. `docs/WORKFLOW-GUIDE.md`
5. `docs/项目使用说明.md`
6. `docs/Agent调度流程.md`

你只要先搞清 3 件事就够了：

1. 这不是游戏源码，而是 Claude Code 的游戏研发协作模板
2. 入口技能是 `/start`
3. 大任务优先交给技能，不优先自己手动点名 Agent

---

## 3. 第一次启动时做什么

在项目根目录启动 Claude Code 后，按这个顺序：

1. 执行 `/start`
2. 根据你的实际情况进入对应路径
3. 不要一上来就让某个 Agent “直接帮我把整个项目做完”

`/start` 会帮你判断你属于哪类情况：

- 完全没想法
- 有模糊想法
- 有清晰概念
- 已经有现成项目

---

## 4. 四种起步路径

## 4.1 路径 A：完全没想法

执行顺序：

1. `/start`
2. `/brainstorm`
3. `/setup-engine`
4. `/map-systems`
5. `/design-system <系统名>`

适合：

- 想从 0 到 1 找游戏方向

## 4.2 路径 B：已有概念，但没做结构化设计

执行顺序：

1. `/setup-engine`
2. `/map-systems`
3. `/design-system <系统名>`
4. `/design-review`
5. `/sprint-plan new`

适合：

- 已经知道自己想做什么
- 想开始系统化推进

## 4.3 路径 C：已有工程或原型

执行顺序：

1. `/project-stage-detect`
2. `/setup-engine`
3. `/reverse-document`
4. `/gate-check`
5. `/sprint-plan new`

适合：

- 手头已经有代码、原型或旧文档

## 4.4 路径 D：只想先学调度方式

执行顺序：

1. 阅读 `docs/examples/README.md`
2. 看一篇 design 示例
3. 看一篇 implementation 示例
4. 看一篇 strategic decision 示例
5. 再自己执行一次 `/start`

适合：

- 你想先理解这个系统“怎么协作”

## 5. 第一轮最推荐你实际跑的命令

如果你只是想尽快上手，我建议你实际做这一组：

1. `/start`
2. `/brainstorm`
3. `/setup-engine godot 4.6`
4. `/map-systems`
5. `/design-system combat-system`
6. `/design-review design/gdd/combat-system.md`

为什么是这组：

- `/brainstorm` 让你感受协作式设计
- `/setup-engine` 让系统知道你用什么引擎
- `/map-systems` 让你看到“从概念到系统拆分”的方法
- `/design-system` 让你看到章节式 GDD 生产
- `/design-review` 让你看到质量关卡怎么工作

## 6. 什么时候该用技能，什么时候该用 Agent

优先级建议如下。

### 先用技能的情况

- 你要做的是一个完整流程
- 你不确定该叫哪些 Agent
- 任务明显跨多个阶段或多个部门

典型例子：

- 从零开始 -> `/start`
- 拆系统 -> `/map-systems`
- 写系统 GDD -> `/design-system`
- 做战斗功能 -> `/team-combat`
- 做 UI 功能 -> `/team-ui`
- 做发布 -> `/team-release`

### 可以直接找 Agent 的情况

- 你知道自己要问的是单一专业问题
- 你已经处于流程中某个明确环节
- 你只需要一个角色的专业判断

典型例子：

- 问创意方向 -> `creative-director`
- 问技术架构 -> `technical-director`
- 问排期与范围 -> `producer`
- 问机制设计 -> `game-designer`
- 问代码架构 -> `lead-programmer`

简化记忆：

> 流程问题找技能，专业问题找 Agent。

## 9. 一次标准会话应该长什么样

一轮健康的会话，通常会出现这些动作：

1. 先读上下文
2. 先问清楚你的目标和约束
3. 给 2 到 4 个方案
4. 由你选方向
5. 出草案
6. 你批准后才写文件
7. 写完后给下一步建议

如果某次会话一上来就：

- 不问背景
- 不给选项
- 不让你确认
- 直接写很多文件

那就偏离了这个仓库的协作原则。

## 10. 如何正确结束一个会话并开启下一个任务

当出现下面几种情况时，建议主动开启一个新对话：

1. 当前任务已经完成
2. 你准备切到明显不同的话题
3. 你需要换一个主要角色或工作流
4. 当前对话已经很长，开始变慢或变乱

一个好的 Agent 不应该只说“换个对话继续”，而应该直接给你可复制的新开场。

标准格式：

```text
[Current task summary]
建议新开对话继续下一个任务，参考开头:
"[Suggested opening prompt with role, task, and prior context summary]"
```

你可以把它理解成“任务交接单”。

例如：

```text
Board-system GDD is complete and reviewed. The next step is prototype planning.
建议新开对话继续下一个任务，参考开头:
"Act as producer. Help me plan Sprint 1 for my Godot 4.6 match-3 project. We already completed the game concept, systems index, and board-system GDD. Focus on MVP scope, prototype tasks, and a 1-week sprint."
```

这会比在一个已经很长的会话里继续跳话题更稳。

## 11. 最容易犯的 8 个错误

1. 把这个项目当成现成游戏源码仓库
2. 不跑 `/start`，直接乱问
3. 不设引擎就开始让 Agent 写实现方案
4. 不先拆系统，直接同时设计很多功能
5. 想让一个 Agent 包办设计、实现、测试、发布
6. 不做 `design-review` 就把 GDD 丢给程序实现
7. 不利用 `team-*`，手动乱串多个 Agent
8. 忽视 hooks 和 rules 给出的提醒

## 12. 最实用的两条建议

### 建议 1：先把设计链跑通一次

哪怕你以后主要写代码，也建议先完整走一遍：

- `/brainstorm`
- `/map-systems`
- `/design-system`
- `/design-review`

因为这是整个模板最核心的骨架。

### 建议 2：跨部门任务优先交给 `team-*`

例如：

- 战斗功能 -> `/team-combat`
- UI 功能 -> `/team-ui`
- 发版流程 -> `/team-release`

这样你能最直观地看到“主会话编排，子 Agent 分工”的真实效果。
