# 三消项目第一周路线图

## 1. 项目定位

你的目标不是机械复刻《开心消消乐》的全部内容，而是：

- 参考它的核心体验
- 做一个属于你自己的原创三消项目
- 用 Claude Code Game Studios 这套模板把项目从概念、系统设计、原型、迭代计划跑起来

建议项目定位：

> 一款面向移动端和 PC 的轻量级关卡制三消游戏，核心循环是“交换相邻方块 -> 形成消除 -> 触发连锁/特效 -> 完成关卡目标”。

重要提醒：

- 可以参考其玩法节奏和商业层面的成熟经验
- 不建议直接复制其名称、角色、美术、音效、关卡数据和 UI 表现
- 最稳妥的做法是：保留“三消”品类核心机制，做自己的主题包装、关卡设计和数值体系

---

## 2. 为什么这个项目适合用这个模板

三消项目非常适合 Claude Code Game Studios，因为它天然需要四类协作：

1. 玩法设计
   - 棋盘规则、消除规则、特效组合、目标类型、失败条件
2. 程序实现
   - 棋盘状态管理、掉落补位、连锁检测、输入交换、动画事件
3. 数值与关卡
   - 步数、目标、障碍物、难度曲线、道具效果
4. 测试与迭代
   - 死局检测、无解重排、边界条件、性能与手感验证

这个模板正好提供：

- `game-designer` / `systems-designer`
- `lead-programmer` / `gameplay-programmer` / `ui-programmer`
- `economy-designer` / `level-designer`
- `qa-lead` / `qa-tester`

---

## 3. 我给你的推荐技术路线

如果你是单人开发，推荐：

- 引擎：Godot 4.6
- 语言：GDScript
- 项目类型：2D
- 开发策略：先做“单关核心循环可玩原型”，再扩展地图、道具、活动和商业化

推荐理由：

- 三消是强规则、强状态驱动、2D 为主的游戏类型
- Godot 对独立 2D 项目很合适
- 你会更快把原型跑起来
- 这个模板对 Godot 也有完整 Agent 支持链

---

## 4. 第一周总目标

第一周不要追求“大而全”，只追求这 5 个结果：

1. 明确游戏概念与差异化方向
2. 完成核心系统拆解
3. 写出最关键的 2 到 4 份 GDD
4. 做出一个可玩的三消核心原型
5. 形成第一轮 Sprint 计划

第一周结束时，你应该至少拥有：

- 一个原创三消项目概念
- 一个系统清单
- 一份核心玩法 GDD
- 一份关卡目标/障碍系统 GDD
- 一个“可交换、可消除、可掉落、可补位”的原型

---

## 5. 第一周详细执行路线

## Day 1：定项目方向，不直接开写代码

### 目标

- 明确“你要做的三消游戏到底是什么”
- 不只说“像开心消消乐”，而是说出你自己的版本

### 建议在 Claude Code 里执行

1. `/start`
2. `/brainstorm 三消 休闲 关卡制 移动端`

### 你要重点回答的问题

- 主题包装是什么
  - 糖果、花园、厨房、旅行、魔法、宠物、餐厅、幻想岛屿？
- 你的差异化是什么
  - 更轻度？
  - 更强调连锁爽感？
  - 更强调障碍解谜？
  - 更强调道具组合？
- 核心目标是哪个阶段
  - 原型验证
  - 小型独立项目
  - 后续商业化移动游戏雏形

### 当天产出

- `design/gdd/game-concept.md`
- `design/gdd/game-pillars.md`

### 推荐 Pillars

你这类项目可以先考虑这 4 个方向：

1. 轻松上手，单局反馈快
2. 连锁触发要足够爽
3. 关卡目标要持续变化
4. 难度上升来自规则组合，不是纯数值堆高

## Day 2：确定引擎和系统地图

### 目标

- 把“一个三消游戏”拆成真正可开发的系统

### 建议执行

1. `/setup-engine godot 4.6`
2. `/map-systems`

### 你的三消项目建议拆出的核心系统

MVP 阶段建议至少包含：

1. 棋盘系统
2. 交换规则系统
3. 匹配检测系统
4. 消除结算系统
5. 掉落与补位系统
6. 连锁系统
7. 关卡目标系统
8. 基础 UI 系统
9. 输入控制系统
10. 关卡配置数据系统

后续阶段再加入：

- 特殊块系统
- 障碍物系统
- 道具系统
- 星级评价系统
- 地图进度系统
- 签到/活动/商业化系统

### 当天产出

- `design/gdd/systems-index.md`
- 明确 MVP 系统优先级

### 当天的关键判断

你要刻意控制范围：

- 第一周不要做地图、签到、商城、活动
- 先做“单局三消”核心循环

---

## Day 3：先写最核心 GDD

### 目标

- 把最关键的三消玩法规则写清楚

### 建议执行

1. `/design-system board-system`
2. `/design-system match-resolution-system`
3. `/design-review design/gdd/board-system.md`
4. `/design-review design/gdd/match-resolution-system.md`

### 这两份文档必须回答的关键问题

#### `board-system`

- 棋盘尺寸固定还是可配置
- 方块类型有几种
- 初始生成是否允许开局自动匹配
- 玩家能否交换任意相邻格
- 什么样的交换才算合法
- 非法交换是否回弹

#### `match-resolution-system`

- 三连、四连、五连、T/L 型怎么判定
- 多组同时匹配时怎么结算
- 消除顺序如何定义
- 连锁如何继续触发
- 动画和逻辑是同步还是事件驱动

### 当天产出

- 2 份通过 review 的核心 GDD

---

## Day 4：设计关卡与特殊玩法骨架

### 目标

- 让你的项目不只是“能消”，而是“能成为关卡游戏”

### 建议执行

1. `/design-system level-goal-system`
2. `/design-system blocker-system`
3. `/design-review design/gdd/level-goal-system.md`

### 推荐先支持的关卡目标

MVP 只做 2 到 3 种：

1. 指定分数
2. 收集指定颜色方块
3. 清除指定障碍

### 推荐先支持的障碍物

MVP 只做 1 到 2 种：

1. 冰块
2. 箱子 / 石块

这样好处是：

- 已经能形成“开心消消乐式关卡感”
- 但实现复杂度还能控住

### 当天产出

- 关卡目标系统 GDD
- 障碍系统初稿

---

## Day 5：开始做原型，不做全套产品

### 目标

- 做出最小可玩版本

### 建议执行

1. `/prototype 三消核心循环`

如果你希望更偏协作式开发，也可以让主线这样走：

1. 让 `lead-programmer` 先设计原型架构
2. 让 `gameplay-programmer` 实现核心逻辑
3. 必要时让 `ui-programmer` 做简易 HUD

### MVP 原型最低要求

必须实现：

1. 生成棋盘
2. 点击/拖拽交换相邻块
3. 检测 3 连及以上
4. 消除
5. 上方掉落
6. 随机补位
7. 连锁继续结算
8. 显示剩余步数和基础目标

可以暂时不实现：

- 特效动画精修
- 地图界面
- 商城
- 活动
- 复杂障碍
- 存档

### 当天产出

- `prototypes/` 下的三消原型
- 原型假设与成功标准

---

## Day 6：验证原型是否成立

### 目标

- 不只看“能跑”，还要看“是否好玩、是否稳”

### 建议执行

1. `/playtest-report`
2. `/code-review`
3. 如涉及关卡难度，执行 `/balance-check`

### 你应该重点验证什么

- 交换是否流畅
- 回弹是否清晰
- 消除判定是否符合直觉
- 掉落和补位是否有爽感
- 连锁是否容易理解
- 有没有死局
- 开局随机性是否太离谱

### 当天产出

- 第一轮试玩反馈
- 原型缺陷清单
- 需要修正的优先项

---

## Day 7：形成正式项目计划

4. 完成基础 HUD
5. 建立关卡数据配置格式
6. 建立死局检测与重排逻辑
7. 写核心自动化测试

### 当天产出

- 第一轮估算
- 范围裁剪结果
- Sprint 1 计划

## 6. 我建议你的系统设计顺序

为了最稳，建议你按这个顺序写 GDD：

1. `board-system`
2. `match-resolution-system`
3. `fall-and-refill-system`
4. `turn-and-goal-system`
5. `special-tile-system`
6. `blocker-system`
7. `level-data-system`
8. `hud-ui-system`

这个顺序的好处：

- 先规则骨架
- 再结算流程
- 再关卡目标
- 最后才是表现与扩展玩法

## 8. 第一周不要做的事

为了保证项目活下来，这几件事建议先别做：

1. 不要一开始就做几十种障碍物
2. 不要一开始就做几十种特效块组合
3. 不要一开始就做地图大世界
4. 不要一开始就做商城和复杂商业化
5. 不要一开始就抄现成关卡
6. 不要先追求美术完成度
7. 不要跳过死局与重排设计

三消最容易踩的坑不是“不会消”，而是：

- 后期规则越来越乱
- 关卡做不出来
- 随机性失控
- 玩起来不爽
